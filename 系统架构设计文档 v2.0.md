一、架构目标（必须明确）
1.1 核心目标
支持复杂策略玩法扩展
支持技能 / Buff / AI 深度系统
支持数据驱动（JSON）
支持战斗回放 / 可复现
支持未来网络同步（可选）
1.2 核心原则（统一思想）
Command驱动 + Simulation内核 + Event解耦 + 数据驱动 + 表现分离
二、总体架构（升级版）
2.1 六层架构（最终形态）
┌──────────────────────────────┐
│ Presentation（表现层）         │ ← MonoGame 渲染/UI
├──────────────────────────────┤
│ Application（应用层）          │ ← Scene / 流程控制
├──────────────────────────────┤
│ Domain（领域层）               │ ← 游戏规则（非战斗）
├──────────────────────────────┤
│ Simulation（模拟层）⭐核心       │ ← 战斗核心（纯计算）
├──────────────────────────────┤
│ Infrastructure（基础设施）     │ ← Event / Command / Tick
├──────────────────────────────┤
│ Data（数据层）                 │ ← JSON配置
└──────────────────────────────┘

三、各层详细设计
3.1 Presentation（表现层）
职责
渲染（SpriteBatch）
UI（CityActionDialog等）
输入捕获（鼠标/键盘）
❗限制（必须遵守）
❌ 不允许写游戏逻辑
❌ 不允许修改数据
✅ 只读数据 + 显示
数据来源
Event → 更新表现
GameState → UI展示
3.2 Application（应用层）
职责
场景管理（Scene）
游戏流程控制
输入转换为Command
核心结构
class GameApp {
    SceneManager sceneManager;
    CommandQueue commandQueue;
}
示例流程
点击按钮 → 转换为Command → 发送到系统
❗关键改造点

👉 现在你是：

点击 → 直接调用Battle

👉 必须改为：

点击 → Command → Simulation执行
