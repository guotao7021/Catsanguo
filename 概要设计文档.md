# 猫三国 - 概要设计文档

> 最后更新：2026-04-15（根据需求文档同步）

## 一、项目概述

### 1.1 项目目标

构建一款参考三国志11的2D策略战斗游戏，具备以下能力：

- 支持事件剧本制开局（选剧本→选势力→开始游戏）
- 支持回合制时间推进（10日/月/季多层迭代）
- 支持世界地图城池争夺（16×10网格）
- 支持3人编队半实时战斗（RTT）+ 武将俘获机制
- 支持外交系统（同盟/通商/停战/劝降）
- 支持内政官员分配制（所有操作需武将执行）
- 支持技能 / Buff / AI 复杂扩展
- 支持数据驱动内容生产（JSON配置）
- 可持续迭代

### 1.2 核心设计原则

**场景驱动 + GameState单例 + 回合制推进 + 数据配置 + 即时渲染**

### 1.3 胜利条件

- **胜利**：玩家势力占领地图上所有城池
- **失败**：玩家势力失去所有城池
- AI势力间同样互相攻伐，遵循相同规则

## 二、总体架构设计

### 2.1 四层架构

| 层次 | 职责 | 核心类 |
|------|------|--------|
| 表现层 | SpriteBatch渲染、Camera2D、字体 | CatSanGuoGame, Camera2D, SpriteSheetManager |
| 场景层 | 游戏流程、场景生命周期 | SceneManager, WorldMapScene, BattleScene |
| 逻辑层 | 战斗引擎、状态管理、AI、回合/外交/俘获 | GameState, EventBus, BuffSystem, SkillTriggerSystem, TurnManager, ScenarioManager, DiplomacyManager, CaptureManager |
| 数据层 | 数据模型、配置表、JSON加载 | DataSchemas, InteriorConfig, DataLoader, ScenarioConfig |

### 2.2 分层职责

**表现层**
- SpriteBatch 2D渲染（预乘Alpha混合）
- Camera2D 世界/屏幕坐标转换
- InputManager 键鼠输入
- FontStashSharp 字体渲染
- 各Renderer绘制组件

**场景层**
- Scene基类（Enter/Update/Draw/Exit）
- SceneManager场景切换（含淡入淡出）
- MainMenuScene：主菜单 + 剧本/势力选择
- WorldMapScene：地图探索/城池管理/军队调度/回合推进
- BattleScene：战斗四阶段（Deploy/Countdown/Fighting/Result）

**逻辑层**
- GameState单例：武将/城池/编队/存档/当前日期/俘虏
- TurnManager：回合推进（10日/月/季多层迭代）
- ScenarioManager：事件剧本加载与初始化
- DiplomacyManager：外交关系管理（同盟/通商/停战/劝降）
- CaptureManager：武将俘获/招降/释放/关押
- EventBus：事件驱动
- BuffSystem：BUFF管理
- SkillTriggerSystem：技能触发链
- BattleContext：战斗上下文
- ArmyManager/MapPathfinder：军队移动和寻路
- FogOfWarManager：雾战
- MoraleSystem/DamageCalculator：战斗计算

**数据层**
- DataSchemas：全部数据类和枚举
- DataLoader：泛型JSON反序列化
- InteriorConfig：建筑/产量配置
- ScenarioConfig：事件剧本配置（scenarios.json）
- UnitCounterConfig：军种克制表
- FormationConfigTable：阵型配置
- CityScaleConfig：城池规模加成

## 三、核心系统设计

### 3.1 场景系统

**设计目标**：管理游戏不同阶段的逻辑和渲染

```csharp
abstract class Scene {
    virtual void Enter();
    virtual void Update(GameTime gameTime);
    virtual void Draw(GameTime gameTime);
    virtual void Exit();
}
```

**场景列表**：
| 场景 | 说明 |
|------|------|
| MainMenuScene | 主菜单，新游戏（含剧本/势力选择）/继续/设置 |
| WorldMapScene | 世界地图，城池交互，回合推进，外交操作 |
| BattleScene | 战斗执行（部署→倒计时→战斗→结算+俘获判定） |

### 3.2 GameState 单例

**设计目标**：全局统一的游戏状态管理

核心能力：
- 当前游戏日期与回合状态管理
- 当前剧本与势力信息
- 武将生命周期管理（历史登录→在野→招募→俘虏→培养）
- 城池资源、建筑和官员分配管理
- 3人编队系统
- 城市占领记录
- 外交关系记录
- 俘虏列表管理
- 人才系统（搜索/说服/招降）
- JSON存档/读档

### 3.3 事件剧本系统

**设计目标**：三国志11式多剧本开局

```
主菜单 → 选择事件剧本 → 选择势力 → 开始游戏
```

- 每个剧本定义一个历史时间点（如"黄巾之乱 184年"、"官渡之战 200年"）
- 剧本包含完整世界初始状态快照：势力/城池/武将/资源/兵力
- 数据从 `scenarios.json` 加载
- ScenarioManager负责剧本解析和GameState初始化

### 3.4 回合制时间系统

**设计目标**：替代30秒连续Tick，改为回合制推进

| 周期 | 触发内容 |
|------|----------|
| **每10日** | 一次回合更新：武将行动、搜索、外交、事件触发 |
| **每1月** | 城池经济迭代：资源产出（仅对已分配内政官的城池生效） |
| **每3月** | 兵力迭代：征兵恢复（仅对已分配军事官的城池生效） |

- TurnManager负责回合推进和多层迭代触发
- 玩家在回合内执行操作，点击"结束回合"推进10日
- 经济/兵力迭代在月/季节点自动触发

### 3.5 战斗系统

**四阶段流程**：
```
Deploy（部署武将）→ Countdown（3秒倒计时）→ Fighting（战斗）→ Result（结算+俘获判定）
```

**核心三件套**：
- **EventBus**：OnBattleStart/OnAttack/OnHit/OnRetreat等事件
- **BuffSystem**：BUFF创建/持续/过期/移除
- **SkillTriggerSystem**：事件驱动技能触发链

**Squad（方阵）模型**：
- 每个Squad对应一个武将+军种+阵型
- AttributeSet属性集 + Modifier加法/乘法修饰
- 自动攻击 + 玩家手动释放技能

**战斗结算与俘获**：
- 武将不会在战斗中死亡
- 失败方武将有概率被俘获（受速度/兵力比/忠诚度影响）
- CaptureManager处理俘获→招降/释放/关押

### 3.6 属性系统（6维）

**武将属性**：

| 属性 | 作用范围 |
|------|----------|
| 武力(Strength) | 战斗攻击力 |
| 智力(Intelligence) | 技能效果、计策成功率 |
| 统帅(Command) | 士气、部队规模上限 |
| 政治(Politics) | 内政效率、建设速度 |
| 经济(Economics) | 城池商业/税收加成 |
| 忠诚度(Loyalty) | 被俘投降概率、叛变概率 |

- 武力/智力/统帅/政治/经济为培养型属性，范围1~100
- 忠诚度为动态属性，范围0~100，受事件/俘虏/待遇影响

**战斗属性计算模型**：
```
最终值 = (基础值 + 固定加成) × 百分比修饰
```

支持来源：装备、Buff、技能、阵型加成

### 3.7 武将特技系统

部分武将初始具备特技（特殊被动能力）：

| 类型 | 说明 |
|------|------|
| 战斗特技 | 战斗中自动触发（如神射：远程伤害+20%） |
| 内政特技 | 内政时提供加成（如耕作：粮食产出+15%） |
| 指挥特技 | 影响部队整体（如鼓舞：士气回复加速） |

### 3.8 Buff系统

**设计目标**：统一所有状态效果

**生命周期**：
```
创建 → 应用效果 → 持续计时 → 过期移除
```

**支持能力**：减速、中毒、眩晕、属性增强等

### 3.9 技能系统

**设计思想**：技能 = 事件触发 + 效果执行

**触发链**：
```
战斗事件(OnAttack) → 检查触发条件 → 执行效果(AddBuff/Damage) → 可能触发更多事件
```

### 3.10 AI系统

**战略AI（大地图）**：
- 攻城优先级判断
- 防守决策
- 内政官员自动分配
- 武将搜索与招募
- 外交决策（结盟/通商/停战/劝降）
- 经济管理（税率/资源调配）

**战斗AI**：BattleAI支持难度等级
- 选择目标（最近/最低血量）
- 技能释放判断（CD检查）
- 撤退决策

### 3.11 内政系统

**核心规则：所有内政操作必须分配武将/官员执行，未分配则不予更新。**

**城池管理**：
- 5种资源：Gold/Food/Wood/Iron/Population
- 建筑5大类：农业/商业/军事/城防/仓储
- 建筑升级：费用 = baseCost × 1.5^(level-1)
- 产量计算：baseProduction × level^1.2 × cityScaleMultiplier
- 经济产出周期：每1月（需分配内政官）
- 兵力恢复周期：每3月（需分配军事官）

**官员分配**：

| 职位 | 关键属性 | 影响 |
|------|----------|------|
| 太守 | 政治+统帅 | 城池总体效率 |
| 内政官 | 政治+经济 | 资源产出加成 |
| 军事官 | 统帅+武力 | 征兵效率+防御 |
| 搜索官 | 智力+政治 | 发现武将概率 |

### 3.12 外交系统

| 操作 | 条件 | 效果 |
|------|------|------|
| 同盟 | 双方无敌对状态 | 共享视野、互不进攻 |
| 通商 | 双方有同盟或中立 | 双方经济收入增加 |
| 停战 | 正在交战中 | 暂停敌对状态（有期限） |
| 劝降 | 对方仅剩1~2城 | 尝试和平吞并弱势势力 |

- DiplomacyManager管理势力间关系
- 外交操作需分配武将执行，受政治和智力影响

## 四、关键基础设施

### 4.1 EventBus

解耦战斗逻辑与表现的事件发布/订阅系统。

### 4.2 数据配置

```
JSON文件 → DataLoader → DataSchemas实例 → GameState运行时数据
```

核心数据文件：scenarios.json / cities.json / generals.json / skills.json / formations.json 等

### 4.3 存档系统

GameState序列化为JSON，保存：当前日期、武将进度、城池进度、编队、占领列表、官员分配、俘虏列表、外交关系。

## 五、表现层设计

### 原则：即时模式渲染

所有UI和游戏元素通过SpriteBatch直接绘制：
- DrawRectangle / DrawString / Draw(texture)
- Camera2D矩阵变换实现世界空间渲染
- 屏幕空间渲染用于HUD和对话框

### 世界地图渲染管线

```
背景 → 州郡 → 地形 → 道路(陆路3px/水路2px) → 城池 → 军队
→ [屏幕空间] HUD → CityActionDialog → 回合信息 → 通知
```

### 战斗渲染

- Squad精灵绘制（位置/动画/血条）
- 浮动伤害数字（FloatingTextManager）
- 技能VFX（环形扩展效果）
- 屏幕震动效果
- 撤退/俘获通知面板

## 七、核心风险与应对

| 风险 | 应对 |
|------|------|
| 数据配置量大 | JSON驱动，可批量生成 |
| 平衡性调整 | 克制表/配置表独立于代码 |
| UI复杂度 | 即时模式渲染，按需绘制 |
| 性能瓶颈 | 2D渲染开销低，SpriteBatch批量绘制 |
| 剧本内容不足 | 优先实现2~3个核心剧本 |
| 武将数据量大 | 分批次配置，先核心武将 |
| 回合制改造风险 | 30秒Tick→月度迭代需全面测试 |

## 八、结论

项目采用 **MonoGame + 场景制 + GameState单例 + 回合制推进 + JSON数据驱动** 的实用架构。当前已实现完整的世界地图→城池管理→军队出征→战斗执行→奖励结算的游戏循环。后续重点是实现事件剧本系统、回合制时间推进、官员分配、外交系统和武将俘获机制，完成从连续Tick到回合制的架构升级。
